# WMM 0.8.1 Playset Scope and Friendly-Fire Visibility Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:executing-plans` to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Restore the no-friendly-fire controls to the primary game-data panel view and make playsets, their active selection, and their load order independent for Warhammer III and Three Kingdoms.

**Architecture:** The frontend receives only the playsets for the selected game; isolation is enforced by SQLite, repository methods that take an explicit game ID, and API call sites that derive the active game from settings. Legacy data is deterministically migrated to Warhammer III. The friend-fire repair is presentation-only: re-order the existing card without altering settings or runtime patch logic.

**Tech Stack:** Python 3.11, SQLite, unittest, Vue 3, Pinia, Vitest, Vite.

## Global Constraints

- Target source version is exactly `0.8.1`.
- Do not run release packaging, create an EXE, change `packaging/update-manifest.json`, create a Git commit, tag, or Release.
- Preserve old playsets and their item order by assigning all legacy rows to `warhammer3`.
- Three Kingdoms starts with its own empty default playset and never receives Warhammer III playsets.
- Keep the original Warhammer III default ID `default`; give each other game a distinct default ID and return `is_default` from the repository.
- Update all six built-in changelog languages with the same release-entry shape.
