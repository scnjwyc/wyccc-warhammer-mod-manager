# WMM 0.8.0 Multi-game, Workshop, and WebView2 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (- [ ]) syntax for tracking.

**Goal:** Add Total War: THREE KINGDOMS support while retaining Warhammer III as the default, harden Workshop publishing ownership, remove the custom update channel, and replace blank-start failure modes with actionable WebView2 diagnostics.

**Architecture:** A GameDefinition registry becomes the source of truth for Steam IDs, executable names, path discovery, launch detection, and feature capabilities. Settings migrate into per-game path records, while API and Steamworks calls obtain the active game's definition instead of using WH3 constants. Startup checks WebView2 before pywebview creates a window, and the static frontend displays a fallback rather than an empty root element.

**Tech Stack:** Python 3.11+, pywebview, Windows registry/ctypes, SteamworksJS bridge, Vue 3, Pinia, Vitest, unittest, Vite.

## Global Constraints

- Modify only source, tests, and documentation inside G:\git\wyccc-warhammer-mod-manager.
- Do not run packaging, create an EXE, write G:\Wyccc's Mod Manager, create a Git tag, publish a Release, or alter packaging/update-manifest.json artifact metadata.
- warhammer3 remains the default selected game; three_kingdoms uses Steam App ID 779340 and Three_Kingdoms.exe.
- Preserve individual game paths across switches. Legacy game_path and workshop_path migrate only into the Warhammer III record.
- Hide and reject only the two requested Warhammer III-specific actions for other games: game-data modification and official launcher-profile import.
- Every new visible string must exist in Chinese, English, Korean, Russian, Japanese, and Spanish.
- Follow TDD: each production behavior is written only after its focused test has failed for the intended missing behavior.
