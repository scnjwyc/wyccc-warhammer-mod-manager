# 0.8.3 Non-ASCII Game Path Launch Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Allow the manager to launch either supported game when its game install or Workshop directory contains Chinese or other non-ASCII characters.

**Architecture:** Keep all stored, scanned, and displayed paths physical. Immediately before generating the game-facing load-order file, map only non-ASCII game and Workshop roots to free per-session Windows `subst` drives, then render the load-order file and launch executable through those ASCII aliases. The process monitor continues to compare the physical executable path because Windows reports the physical image path for a process started via `subst`.

**Tech Stack:** Python 3.11+, `pathlib`, `ctypes` Win32 drive enumeration, Windows `subst.exe`, existing unittest suite.

```python
class LaunchPathAliases:
    def prepare(self, game_path: str, workshop_path: str) -> LaunchPathMap:
        roots = [Path(game_path), Path(workshop_path)]
        aliases = [(root, self._alias_for(root)) for root in roots if _needs_alias(root)]
        return LaunchPathMap(tuple(aliases))
```

Use `GetLogicalDrives` to reserve only free drive letters, invoke `subst.exe <letter>: <physical-root>` with a list argument vector, cache successful aliases for the process lifetime, and raise a clear `ValueError` if no drive can be assigned.
