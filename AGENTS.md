# Wyccc's Mod Manager 项目规则

- 只有使用 GPT5.6 Sol 以外的模型时才使用 superpower。
- 修改本项目时，只更新 `G:\git\wyccc-warhammer-mod-manager` 内的源码、测试和文档。
- 默认不得运行发布打包流程，也不得写入 `G:\Wyccc's Mod Manager`；只有用户明确要求打包或发布某个版本时，才视为对该版本的打包脚本、发布目录、Git 提交、Tag 和 Release 操作的授权。
- 获得版本发布授权后，只能使用项目内既有发布脚本，并且只处理该版本所需的源码、测试、文档、正式清单和 `G:\Wyccc's Mod Manager\Wyccc's Mod Manager.exe`，不得把发布范围扩大到其他用户文件。
- 任何设计文本的修改，都需要同步修改多语言文件，确保所有内置语言的对应文本保持一致。

## 正式版本发布流程

1. 发布前检查
   - 确认应用版本号、打包版本信息和更新日志均为目标版本，所有内置语言中的版本与更新日志保持同步。
   - 更新日志必须简洁明了，只介绍用户可见的功能与修复，不描述实现细节。
   - 更新日志统一按“新增”“调整”“修复”三类归类；每个版本的中英文及其他内置语言条目必须使用同一分类结构，且仅记录用户可见的变化。
   - 更新日志不得写入捐赠二维码或捐赠图片信息。
   - 检查工作树和待提交文件，不得提交 AI 密钥、API Token、`.env`、本机设置数据库、日志、缓存或包含用户私有路径的文件。
2. 测试与打包
   - 运行后端测试、前端测试、前端构建和 `git diff --check`。
   - 使用 `scripts\package_release.ps1` 或 `一键打包发布版.cmd` 进行正式打包，输出目录固定为 `G:\Wyccc's Mod Manager`，正式附件只使用 `Wyccc's Mod Manager.exe`。
   - 校验 EXE 的产品名称、文件版本、产品版本、文件大小和 SHA-256。
3. 正式更新清单
   - 使用打包结果的真实文件大小和 SHA-256 更新 `packaging\update-manifest.json`。
   - 清单中的版本、发布日期、各语言更新日志、下载地址、文件名、大小和 SHA-256 必须与本次发布一致。
   - 重新运行与版本、清单和更新检测有关的测试。
4. Git 提交与 Tag
   - 提交正式版本变更，提交信息使用 `release: <版本号>`。
   - 创建与版本号同名的 annotated Tag，并确认 Tag 指向正式版本提交。
   - 将分支分别推送到 GitHub 的 `main` 和 Gitee 的 `master`，并把同一个 Tag 推送到两个仓库。
5. 创建双平台 Release
   - GitHub：创建公开、非草稿、非预发布的 Release，标题使用 `Wyccc's Mod Manager <版本号>`，正文使用该版本正式更新日志，并上传 EXE。
   - Gitee：Codex 只负责打开创建页、选择已有 Tag、填写标题和完整更新日志；用户负责点击附件上传、选择 EXE，并点击“创建发行版”。Codex 不代替用户执行 Gitee 的附件上传和最终确认。
   - 如果 Gitee Release 被中断或删除，Codex 重新填写表单后再次交由用户完成上传和确认，后续版本沿用这一分工。
6. 发布后验证
   - 通过两个平台的公开 Release 页面或 API 核对版本号、标题、发布状态和附件。
   - 实际下载 Gitee 的 EXE，核对文件大小和 SHA-256；通过 GitHub API 核对附件大小、上传状态和 digest。
   - 核对两个仓库的远端分支、Tag 和公开更新清单，确保版本、下载链接、大小和 SHA-256 一致。
   - 最终向用户汇报提交、Tag、两个 Release 链接、EXE 路径、文件大小、SHA-256 和测试结果。
