# Wyccc's Mod Manager

[简体中文](README.md) | **English**

A lightweight mod manager that supports both Total War: WARHAMMER III and Total War: THREE KINGDOMS.

The current version is `0.8.3`. It brings mods, playsets, the Steam Workshop, and game launching together in one interface, with no complicated setup required.

## Features

- Easy to set up and ready to use without complicated configuration.
- Supports Total War: WARHAMMER III by default and Total War: THREE KINGDOMS through the game selector, with separately saved and automatically detected game paths.
- Manage enabled and disabled mods at a glance. Supports Ctrl multi-selection, Shift range selection, batch drag-and-drop sorting, and direct movement between both lists.
- Ctrl+A selects the current list, or only visible search results while searching. Low-overhead filesystem events keep Data and Workshop additions/removals in sync and can be disabled in Settings.
- Automatic sorting helps new users avoid having to manage load order manually.
- Playset-based mod lists, similar to Paradox games, let you save different mod combinations. Every enable, disable, or sorting action immediately updates the active playset.
- Quickly search, filter, and organize mods, and save aliases, notes, and custom categories. AI can also generate mod aliases and summaries automatically.
- Powerful publishing tools make it easier for modders to publish and update their own mods. Mods maintained in multiple languages no longer risk having one language overwrite another after publishing.
- Share or import playsets. You can send your current mod list to friends for multiplayer, and they can import it and automatically subscribe to missing mods.
- Automatically checks for missing dependencies and mod version issues. Unnecessary warnings can be ignored per mod, such as false missing-dependency warnings for translation mods.
- Launch the game directly, continue the most recent campaign, or load a selected save from the save list.
- Compare the MODs used by a save and enable them in one action.
- Import presets from the official game launcher to quickly create a new playset or replace the current one.
- Dynamically adjust unit sizes together with formation rank depth; regular single-entity monsters can adjust by health or scale, and unit and spell friendly fire can be disabled, with compatibility across all MODs and game updates!
- Copy MOD paths or safely move MOD files to the Recycle Bin from the context menu. Unsubscribing also cleans the matching Workshop folder, and these file operations are blocked while the game runs.
- Automatically switches to a static low-consumption page while the game runs, pausing scanning and directory monitoring until the game exits.
- Automatically checks for new versions and lets you view release notes, download updates, and install them from within the application.
- Built-in support for Simplified Chinese, English, Korean, Russian, Japanese, and Spanish.

## Interface and Architecture

The main interface uses a three-column layout: the selected mod's information on the left, disabled mods in the middle, and the ordered enabled-mod list on the right. The details panel shows the name, source, author, creation and update times, path, basic Workshop information, description, alias, and notes.

| Layer | Technology and responsibility |
| --- | --- |
| Frontend | Vue 3, Pinia, and Vite; provides the three-column interface, list interactions, playsets, and settings window |
| Desktop container | pywebview; loads the built frontend and exposes a single RPC entry point to the page |
| Backend | Primarily the Python standard library; handles Steam path discovery, Pack scanning, load manifests, launching, and local persistence |
| Workshop localization bridge | Node.js + steamworks.js; like WH3 Mod Manager, directly calls the local Steamworks UGC API to query a selected language |
| Settings and cache | Atomically written JSON files: `settings.json` and `workshop_cache.json` |
| User state | SQLite; stores playsets and enabled order, aliases, notes, multi-select categories, hidden state, Workshop IDs associated with local mods, and manifest backup indexes |

Main directories:

```text
backend/          Python backend and explicit RPC interface
frontend/         Vue frontend
frontend/dist/    Production frontend assets (generated by the build)
tests/            Python tests
licenses/         Original licenses for referenced projects
main.py           Desktop application entry point
```

## Development

Requirements:

- Windows 10/11 (currently the primary target platform)
- Python 3.11 or later
- Node.js 22 or later
- pnpm 11
- The Steam edition of Total War: WARHAMMER III or Total War: THREE KINGDOMS installed for real scanning and launching

### Windows One-Click Entry Points

Double-click either file in the project root:

- `一键启动管理器.cmd`: Automatically prepares Python and frontend dependencies, builds the latest frontend, and launches the pywebview desktop window. The application does not provide a browser mode.
- `一键打包发布版.cmd`: Runs backend and frontend tests, builds the frontend, and produces the single release file `Wyccc's Mod Manager.exe`. The default output directory is `G:\Wyccc's Mod Manager`.

Both entry points share the project's `.venv-build` environment. The first run may need to download dependencies, so Python 3.11+, Node.js 22+, and pnpm 11+ must already be installed. After packaging succeeds, the window remains open and displays the result. A clean release directory contains only `Wyccc's Mod Manager.exe`.

To change the packaging directory, set `WMM_OUTPUT_DIR` before running the packaging entry point:

```powershell
$env:WMM_OUTPUT_DIR = "D:\WMM-Release"
.\一键打包发布版.cmd
```

Install Python development dependencies:

```powershell
cd G:\git\wyccc-warhammer-mod-manager
python -m venv .venv
.\.venv\Scripts\Activate.ps1
python -m pip install -U pip
python -m pip install -e ".[dev]"
```

Install frontend dependencies:

```powershell
cd G:\git\wyccc-warhammer-mod-manager\frontend
pnpm install
```

Use two terminals during development. Start Vite first:

```powershell
cd G:\git\wyccc-warhammer-mod-manager\frontend
pnpm dev
```

Then start pywebview and point it to the Vite URL:

```powershell
cd G:\git\wyccc-warhammer-mod-manager
.\.venv\Scripts\Activate.ps1
python main.py --dev-url http://127.0.0.1:5173
```

Use `--data-dir <directory>` to temporarily override the application data directory for development and isolated testing.

## Testing and Build

Run backend and frontend tests separately:

```powershell
cd G:\git\wyccc-warhammer-mod-manager
python -m pytest

cd frontend
pnpm test
```

The project build entry point performs checks and generates production frontend assets:

```powershell
cd G:\git\wyccc-warhammer-mod-manager
python scripts/build.py
```

After the build completes, start the application from the project root:

```powershell
python main.py
```

To create a distributable package, install the build dependencies and use the optional packaging entry point:

```powershell
python -m pip install -e ".[build]"
python scripts/build.py --package
```

Use `--output-dir <directory>` to specify the release destination, for example:

```powershell
python scripts/build.py --package --output-dir "G:\Wyccc's Mod Manager"
```

The packaged application should still be manually verified in a real Total War: WARHAMMER III installation. This project currently does not provide an installer.

## Local Data, Privacy, and Networking

The application does not require a separate manager account and includes no telemetry. It does not upload local Pack files except when the user explicitly confirms “Upload to Workshop” or “Update on Workshop” in the publishing window. Load order, aliases, notes, and categories always remain on the local computer. When running from source, the default Windows application data directory is:

```text
%APPDATA%\WycccModManager\
```

The PyInstaller release contains only the EXE. On first launch, the application creates a `data/` directory beside the EXE for portable data. If the release directory is not writable, it falls back to the user directory above. The `--data-dir` command-line option and `WYCCC_MM_DATA_DIR` environment variable can explicitly override this behavior. During an upgrade, if no data exists in the new location, the application checks the legacy `%APPDATA%\WycccWarhammerManager\` and `%APPDATA%\WycccWarhammerModManager\` directories in order so existing settings and playsets are not lost. Keep the EXE in its own application folder and create a Desktop shortcut; placing the EXE directly on the Desktop also places `data/` there by design.

The data directory may contain:

- `settings.json`: Paths and scanning options.
- `state.db`: Playsets and enabled order, personal labels, mod categories, and local Workshop associations.
- `workshop_cache.json`: Previously retrieved public Workshop metadata.
- `backups/`: Load-manifest backups created before saving.
- `updates/`: Verified update EXEs, replacement scripts, and failed rollback logs.
- `logs/app.log`: The manager's own runtime log.

When background refresh on startup is enabled, or when “Refresh Workshop Information” is selected, the application retrieves English default information in batches from Steam's public `GetPublishedFileDetails` endpoint. It then uses the local Steamworks UGC API to supplement the title and full description in the active language. Results are cached per language in `workshop_cache.json`. Legacy web-scraping caches are rebuilt through Steamworks during the first refresh to avoid Steam Community rate limits. Workshop pages are opened in the system browser. When publishing a user-owned mod, only the selected Pack, preview image, title, description, tags, visibility, and update notes are sent to the Steam Workshop. Preview images must be PNG or JPEG and no larger than 1 MB.
